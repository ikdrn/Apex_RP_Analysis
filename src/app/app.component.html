この設計書は、非エンジニアやジュニアエンジニアでも読めるように「目的→仕組み→テスト」の順でまとめています。

1. 要件定義書

  • 背景・目的: RP推移を誰でも見える化し、日々の変化を早く判断できるようにする。
  • スコープ: Web画面（グラフ/表/CSV）とAPI（取得・期間指定・基本的な保護）。
  • 業務フロー: As-Is=手作業確認、To-Be=画面アクセス→期間選択→確認→必要ならCSV出力。
  • 機能要件: 期間切替、再取得、一覧表示、CSV、エラー表示、設計書閲覧。
  • 非機能要件: APIの安定性（レート制限）、可用性（Vercel）、セキュリティ（CORS/任意Basic認証）。
  • 外部I/F要件: Supabase REST APIとHTTPS通信。
  • データ移行要件: 既存の `player_rp` をそのまま利用（移行処理なし）。
  • 導入・運用要件: Vercel環境変数設定とログ監視、必要時に設定更新。

2. 基本設計書

  • システム構成図: Browser → Vercel(Angular + API) → Supabase。
  • 画面遷移図: 同一画面内タブ遷移（Analysis / Data Table / 設計書）。
  • 画面レイアウト: ヘッダー、操作ボタン、統計カード、メインタブ領域。
  • 帳票レイアウト: CSV（id, rp, created_at）。
  • テーブル定義/ER: `player_rp(id, rp, created_at)` を参照。1テーブル構成。
  • 外部連携仕様: GET `/api/get-rp?days=7|30|90` → Supabase REST呼び出し。
  • 業務ロジック一覧: 集計表示（最新/最大/最小/変化量）、期間フィルタ。
  • 権限・ロール: 基本は閲覧者、必要ならBasic認証で保護。

3. 詳細設計書

  • モジュール/クラス: `AppComponent`（UI制御）、`api/get-rp.js`（API処理）。
  • シーケンス: 画面起動→AngularがAPI呼び出し→APIがSupabase参照→結果表示。
  • 関数仕様: `loadRecords`, `onRangeChange`, `refresh`, `downloadCsv`, API `handler`。
  • 詳細ロジック: days検証（7/30/90）、失敗時エラー文言、成功時グラフ更新。
  • 共通部品: Chart.js / ng2-charts / Angular HttpClient。
  • DB物理設計: 既存DB利用。将来は `created_at` インデックス強化を推奨。
  • エラーハンドリング/ログ: APIで`console.error/warn/info`、画面に説明文表示。
  • APIリファレンス: GET `/api/get-rp`（query: `days`、response: RPレコード配列）。

4. 各テスト仕様書

  • UT: 関数単位（期間切替、CSV生成、エラー分岐、境界値: days不正値→30）。
  • IT: 画面→API→DB連携、タブ遷移、連続操作（切替→更新→CSV）。
  • ST: 全体動作、ビルド健全性、レスポンス遅延時表示、ログ確認。
  • 証跡: `docs/test-evidence.md` と `docs/evidence/*.txt` に保存。
